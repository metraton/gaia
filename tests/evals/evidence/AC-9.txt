AC-9 — Fix the FALSE POSITIVE of the `heuristic-long-code` classifier
=====================================================================
Date: 2026-06-27
Branch: harden/approval-grants-m1-loop
Status: DONE — long, PROVABLY read-only `python3 -c "..."` inline code
        (import + SELECT/PRAGMA + print + local assignments, no write call) is
        no longer flagged MUTATIVE/T3 by length alone; every mutating payload
        (open-write, os.system, subprocess, AND AST-clean sqlite INSERT/UPDATE
        via bound-method execute/commit) STILL classifies T3. No false negatives.

The gap
-------
hooks/modules/security/mutative_verbs.py, _layer3_length_check():
  When the Python AST analyzer (inline_ast_analyzer.analyze_python_inline)
  parsed a payload cleanly and found NO dangerous CALL, the caller fell through
  to the length heuristic (3b). That heuristic flags ANY inline code whose
  code_portion exceeds MAX_NORMAL_INLINE_LENGTH (500 chars) as MUTATIVE/T3 —
  by LENGTH, regardless of content. A long but purely read-only payload
  (open DB, run SELECT/PRAGMA, print rows) was blocked as if it mutated.

Why the obvious "exempt anything AST-clean" fix would have OPENED A HOLE
-----------------------------------------------------------------------
analyze_python_inline is a BLOCKLIST: a clean result means "no KNOWN dangerous
call in the catalog", which is NOT the same as read-only. Mutations the
blocklist cannot see statically parse cleanly yet mutate:
  * cur.execute("INSERT ..."), con.execute("UPDATE ..."), con.commit()
    -> `execute`/`commit` are bound methods on an arbitrary object, not a
       dotted module call in _DANGEROUS_CALLS.
  * os.environ['X'] = v, sys.path = [...]  -> __setitem__/__setattr__, no Call.
Today the LENGTH heuristic is the ONLY thing catching a long version of these.
Exempting all AST-clean long code would have let a 600-char INSERT/commit run
without approval. Confirmed by repro: the long AST-clean sqlite-write payload
is is_mutative=True verb=heuristic-long-code BEFORE the fix, and must STAY so.

The fix — a positive read-only allowlist, the inverse discipline
----------------------------------------------------------------
hooks/modules/security/inline_ast_analyzer.py
  New public fn is_provably_read_only_python(code) + class _ReadOnlyChecker.
  Returns True ONLY when EVERY statement and EVERY call is on a positive
  read-only allowlist; returns False on ANY uncertainty (parse failure too).
  Contract: a mutation must NEVER classify read-only, even at the cost of
  leaving some genuine reads length-flagged (those stay T3-APPROVABLE — never
  silently run).
  Allowed: Import/ImportFrom, Expr, Assign/AnnAssign/AugAssign to LOCAL Name/
    Tuple targets only (Subscript/Attribute targets rejected — they can mutate
    via __setitem__/__setattr__), For/While/If/With/FunctionDef/Return/Pass/
    Break/Continue/Assert. Calls: read-only builtins (_READ_ONLY_BUILTINS, no
    open/exec/eval/compile/__import__/getattr/setattr), read-only methods
    (_READ_ONLY_METHODS: fetchone/fetchall/keys/get/...), a tiny set of
    module-level read constructors (_READ_ONLY_DOTTED_CALLS: sqlite3.connect,
    json.dumps, os.getcwd, ...). execute/executemany/executescript are allowed
    ONLY when the first arg is a STRING LITERAL whose leading token is in
    _READ_ONLY_SQL_PREFIXES (select/pragma/explain/with/values/show) AND which
    contains NO _MUTATING_SQL_KEYWORD anywhere (defeats CTE-wrapped DELETE,
    stacked SELECT;DROP, and non-literal SQL via variable/f-string/concat).
  Rejected by construction: del, class/try/global, exec/eval/getattr dispatch,
    write_text/os.remove/subprocess/open-write, commit(), attribute/subscript
    assignment, any unrecognized node/call/target.

hooks/modules/security/mutative_verbs.py
  _layer3_length_check(): before flagging on length, for Python interpreters
  only, re-extract the payload and exempt iff is_provably_read_only_python is
  True -> returns READ_ONLY verb="inline-code-readonly". Non-Python payloads
  (node/ruby/perl/...) are NEVER exempted (no AST). Import wired with a
  defensive fallback (_is_provably_read_only_python = None on ImportError).

Verification — classifier (hooks.modules.security.mutative_verbs)
-----------------------------------------------------------------
Measured via detect_mutative_command(cmd). Read-only body = 509 chars of code
(import sqlite3 + sqlite3.connect + PRAGMA + SELECT + print + close), > 500.

BEFORE (the false positive):
  long read-only python3 -c (509 chars)        -> True  MUTATIVE  heuristic-long-code   "unusually long (509 chars > 500 limit)"

AFTER (fixed):
  long read-only python3 -c (509 chars)        -> False READ_ONLY inline-code-readonly  "long (509 chars) but provably read-only"

NO REGRESSION — every mutating payload STILL T3 (is_mutative=True):
  python3 -c "open('x','w').write('y')"        -> True  MUTATIVE  open-write          (AST: FILE_WRITE)
  python3 -c "import os; os.system('rm -rf /')"-> True  MUTATIVE  os-system           (AST: PROCESS_EXECUTION)
  python3 -c "import subprocess; subprocess.run(['rm','x'])" -> True MUTATIVE subprocess-run
  long AST-CLEAN sqlite write (629 chars):
    import sqlite3; con=...; con.execute('INSERT INTO t...'); con.commit(); pad
                                               -> True  MUTATIVE  heuristic-long-code  <-- HOLE STAYS CLOSED
  long node -e (510 chars, non-Python)         -> True  MUTATIVE  heuristic-long-code  (exemption is Python-only)
  long sqlite write via VARIABLE (non-literal)  -> True  MUTATIVE  heuristic-long-code

Adversarial allowlist battery (is_provably_read_only_python directly): 31/31
correct, 0 false negatives. Mutation vectors that must NOT be exempted and were
not: INSERT/UPDATE/DELETE/DROP literals, CTE-wrapped DELETE, stacked
SELECT;DROP via executescript, bare commit(), non-literal/f-string/concat SQL,
open-write via var, os.environ setitem, attr/subscript assignment, del, exec,
eval, getattr dispatch, __import__, unknown local func call, write_text,
os.remove, class def, try/except, global, augassign to subscript.
