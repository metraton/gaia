What the guard does
-------------------
A reusable, table-agnostic guard that asserts every test-DB fixture which
hand-rolls a CREATE TABLE for a production table declares a column shape
consistent with the production schema (gaia/store/schema.sql). It fails on:
  - MISSING column (present in production, absent from fixture) — the
    OperationalError-in-waiting this guard exists to catch
  - type / NOT NULL / DEFAULT shape drift on shared columns
  - stale allowlist entries (a documented intentional-divergence that is no
    longer actually divergent)

Mechanism: materializes gaia/store/schema.sql in-memory (PRAGMA table_info) for
the production spec, extracts each fixture's CREATE TABLE (works for both .sql
files and DDL embedded in Python string literals, via balanced-paren scan),
introspects it the same way, and diffs the two column specs. New (fixture, table)
pairs are added to FIXTURES_UNDER_AUDIT — no logic changes needed per fixture.
