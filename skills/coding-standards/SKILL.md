# Coding Standards

Code is read far more often than it is written, and it outlives the session,
ticket, or conversation that produced it. These are the language-agnostic
rules for writing code so that a future reader — human or agent, with none of
today's context — can trust what they see without a second source.

## Quick-scan

| Concern | Rule |
|---------|------|
| Module purpose | One doc-header per module: state its purpose, key inputs, and gotchas. The single place a reader learns "why this exists." |
| Inline comments | Only for the non-obvious "why." Never narrate what the code already says. |
| Rationale placement | Each rationale lives in exactly one place. A second file references it; it never repeats it. |
| Input documentation | Lives in the language's native doc mechanism (see `reference.md`), never duplicated in a plain comment beside it. |
| Dead code | Delete it. Git history is the record, not a comment. |
| Temporal / attribution comments | None. No "added by X on date," no changelog-in-comment — `git blame` and the commit message already own that. |
| Tooling / plan traces | Zero. No task IDs, AC numbers, Finding/Risk labels, "generated by," or any pointer to the process that produced the code. |
| Style | Consistent within a module and a repo. Match the existing pattern before introducing a new one. |
| Length vs. redundancy | Length is not the enemy; redundancy is. A comment earns its line only if it says something the code cannot. |

## The doc-header

Every module gets exactly one doc-header, placed at its top, that is the
single source of the module's "why": what it exists to do, what inputs it
depends on that are not obvious from the signature, and any gotcha a caller
would otherwise discover the hard way. If that rationale is worth stating
anywhere, it is worth stating here once — not repeated at every call site
that happens to touch the module.

## Inline comments: the why-not-what test

Before writing a comment, ask whether the next line already says it. If the
code reads clearly on its own, a comment restating it adds a second thing to
keep in sync and drops the first time it drifts.

```python
# narrates — delete
# loop over items
for item in items:
    process(item)

# earns its place — states the non-obvious constraint
# insertion order matters: downstream dedup relies on last-write-wins
for item in items:
    process(item)
```

## No tooling, AI, or plan-system traces

A comment that references the process that produced the code — a task ID, an
acceptance-criterion number, a "Finding" or "Risk" label, "generated by," "as
requested," or any other pointer to a ticket, plan, or session — means
nothing to a reader who does not have that ticket open. The commit message,
the PR, and the issue tracker already carry that provenance; code comments
are not a second place for it. Strip the process pointer; keep only the
durable rationale, if any survives once the pointer is gone.

## Redundancy is the threshold, not a line count

A rationale said once is documentation. Said in two places, it is drift
waiting to happen — the two copies will not update together. Before adding a
comment, check whether it repeats the doc-header, the function's own
docstring/JSDoc, or a comment already a few lines up; if it does, delete one
copy rather than keeping both "just in case." This is a judgment call made at
review time, not a fixed character or line-count rule.

## Not in scope here

Project-specific numeric knobs — a repo's own line-length limit, a specific
timeout constant, a lint threshold — are that project's configuration, not a
coding-standards rule. This skill teaches the principle; the number, if any,
lives in the project's own lint config or style guide.

For the per-stack table of where input documentation natively lives
(Terraform `description`, JSDoc, Python docstrings, and others), see
`reference.md`.
